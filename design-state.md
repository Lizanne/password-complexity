## Brief Summary
- **Problem:** Players need to create strong passwords without abandoning the form. Feedback should teach as the user types — not punish at submit.
- **Primary persona:** New players registering an account (mobile-heavy, distracted, mixed digital literacy)
- **Secondary persona:** Existing players doing password reset (possibly stressed/locked out) or change in settings
- **Success metric:** Drop-off at password step measurably lower than current baseline; zero post-submit complexity rejections

## Design Principles
*(Set by design-strategist, 2026-05-26 — see `docs/designpowers/strategy.md` Section 1 for full definitions including "what it rules out" and "how to test it")*

**P1: Teach as they type, never punish at submit**
Every rule is visible from focus. Submit is gated at "Good" or above — post-submit complexity rejections are architecturally impossible.

**P2: The same verdict through every channel**
Strength tier communicated simultaneously via colour, segment count, text label, and screen-reader live region. No information channel is a subset of another.

**P3: Show only what matters for the current task, then get out of the way**
Unmet rules are prominent. Met rules quiet down. *(Note: checklist itself stays visible always — see user override 2026-05-26. Quieting happens within the list, not by collapse.)*

**P4: Calm the anxious user, don't alarm them**
Amber (not red) for Weak/Fair. Copy in second-person present tense. No words like "invalid," "failed," "rejected." Micro-animations ease, never startle.

**P5: One component, context-aware — never three separate implementations**
A single component with a `context` prop (`registration | reset | settings`). Only copy changes between contexts. Rules, scoring, behaviour, and accessibility are identical.
